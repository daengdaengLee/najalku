# eBPF 훅 — XDP, tc, socket 의 부착 위치와 권한

이 문서는 Cilium 시리즈의 두 번째 자료입니다. 이전 세션(0007 eBPF 기초)은 eBPF 가 hook point, verifier, JIT, BPF map 의 조합이라는 점을 설명하면서, Cilium 이 사용하는 세 hook(XDP / tc / socket)을 한 줄씩만 소개했습니다. 이 문서는 그 세 훅이 커널 네트워크 스택의 정확히 어디에 부착되고, 각각 어떤 컨텍스트에 접근하며, 어떤 처리 권한을 갖는지를 한 그림 위에서 비교합니다.

## 문제 — 0007 이 남긴 질문

0007 은 Cilium 이 사용하는 hook 으로 XDP, tc ingress/egress, socket 세 가지가 있다는 점만 짚고 끝냈습니다. 그러나 Cilium 이 같은 패킷에 대해 서로 다른 책임을 서로 다른 훅에 분담시키는 이유를 이해하려면, 각 훅이 NIC 와 애플리케이션 사이 어느 단계에 끼어드는지, 그 단계에서 어떤 데이터(raw 패킷 / `sk_buff` / 소켓 메타데이터)를 다룰 수 있는지, 그리고 훅이 어떤 결정 권한을 가지는지를 정렬해서 봐야 합니다. 같은 "hook" 이라는 이름을 쓰지만 실제로는 셋이 부착 자리도, 접근 가능한 컨텍스트도, 결정 권한도 각기 다르기 때문입니다.

## 핵심 한 줄

세 훅은 NIC 와 애플리케이션 사이의 서로 다른 단계에 부착되어, 서로 다른 컨텍스트(raw 패킷 / `sk_buff` / 소켓 메타데이터)와 서로 다른 결정 권한을 가진 채로 패킷의 흐름에 개입합니다. 부착 자리가 다르면 그 훅이 할 수 있는 일과 들이는 비용이 달라집니다.

## 한 그림으로 보는 부착 위치

세 훅의 자리를 NIC → 애플리케이션 경로 위에 한꺼번에 표기하면 차이가 한눈에 드러납니다.

```
   ────── 수신 경로 ──────                    ────── 송신 경로 ──────

        NIC                                         애플리케이션
         │                                                │
         ▼                                                ▼
    [XDP hook]   ◀── raw 패킷, sk_buff 이전           [socket hook] ◀── cgroup, connect()/sendmsg()
         │                                                │
         ▼                                                ▼
      sk_buff 할당                                     소켓 계층
         │                                                │
         ▼                                                ▼
    [tc ingress] ◀── sk_buff, qdisc 진입              네트워크 스택
         │                                                │
         ▼                                                ▼
      네트워크 스택                                  [tc egress]   ◀── sk_buff, qdisc 출구
         │                                                │
         ▼                                                ▼
       소켓 계층                                         NIC
         │
         ▼
      애플리케이션
```

XDP 는 가장 위(NIC 직후), socket 은 가장 아래(애플리케이션 직후)에서 패킷에 개입하고, tc 는 그 사이의 `sk_buff` 단위 처리 구간을 양쪽에서 받칩니다. 같은 패킷이 여러 훅을 차례로 지나갈 수 있고, 각 훅에서 패킷을 통과시키거나 끊거나 다른 곳으로 돌릴 수 있습니다.

## 메커니즘 — 훅별 상세

각 훅을 같은 네 가지 축으로 비교합니다. 부착 위치(커널 어디인지), 컨텍스트(어떤 데이터를 보는지), 결정 권한(무엇을 결정할 수 있는지), 그리고 Cilium 이 그 자리에서 무엇에 쓰는지입니다.

### XDP — NIC 직후, raw 패킷

XDP(eXpress Data Path)는 NIC 드라이버가 패킷을 받아들이는 가장 이른 지점에 부착됩니다. 패킷이 아직 `sk_buff`(소켓 버퍼) 구조체로 변환되기 전이라, 커널이 패킷 메타데이터를 채우는 비용이 발생하지 않습니다. 일부 NIC 드라이버는 BPF 프로그램을 드라이버 안에서 직접 호출(native XDP)하고, 그렇지 않은 경우에는 커널이 `sk_buff` 를 만든 뒤 실행되는 generic XDP 폴백으로 동작합니다. 이 경우 실제 실행 시점은 `sk_buff` 생성 이후가 되어 비용이 발생하지만, 의미상 같은 hook 으로 처리되어 스택 본격 진입 직전이라는 위치는 동일합니다.

XDP 프로그램은 `xdp_md` 컨텍스트를 받습니다. 이 컨텍스트는 패킷의 raw 바이트에 직접 접근할 수 있는 포인터(`data` 와 `data_end`)를 포함합니다. 즉 XDP 는 패킷을 바이트 수준으로 검사·수정할 수 있습니다.

XDP 가 반환할 수 있는 액션은 네 가지입니다.

- `XDP_DROP`: 패킷을 즉시 폐기합니다. 스택 진입 비용이 발생하지 않습니다.
- `XDP_PASS`: 패킷을 일반 네트워크 스택으로 넘깁니다.
- `XDP_TX`: 같은 NIC 로 패킷을 되돌려 보냅니다.
- `XDP_REDIRECT`: 다른 NIC 또는 다른 CPU 로 패킷을 전달합니다.

Cilium 은 XDP 의 빠른 위치를 부하 분산 가속 경로에 활용합니다. 구체적인 LB 시나리오는 이후 Service LB 세션에서 다룹니다.

### tc ingress/egress — `sk_buff` 단위

tc 훅은 트래픽 제어(traffic control) 계층의 qdisc·class·filter 구조 위에 BPF 필터로 부착됩니다. 각 네트워크 디바이스(물리 NIC 이거나 Pod 의 가상 인터페이스)에 ingress 와 egress 양쪽으로 붙을 수 있습니다. ingress 는 디바이스로 들어온 패킷이 라우팅 등 본격적인 네트워크 스택 처리를 만나기 전에 실행되고, egress 는 패킷이 NIC 로 송출되기 직전에 실행됩니다.

tc 훅이 받는 컨텍스트는 `__sk_buff` 입니다. `sk_buff` 가 이미 만들어져 있으므로 인터페이스 인덱스, 마크, 그 밖에 헬퍼로 접근 가능한 메타데이터를 함께 다룰 수 있습니다. XDP 보다 늦은 위치이지만 라우팅·필터링이 본격적으로 일어나기 전이라는 점은 동일합니다.

tc 훅의 반환 액션은 다음과 같습니다.

- `TC_ACT_OK`: 다음 처리 단계로 패킷을 보냅니다.
- `TC_ACT_SHOT`: 패킷을 폐기합니다.
- `TC_ACT_REDIRECT`: 패킷을 다른 인터페이스로 전달합니다.

Cilium 은 tc 훅을 보안 정책(L3/L4) 집행에 사용합니다. 정책 결정의 자세한 메커니즘은 NetworkPolicy 세션에서 다룹니다.

### socket — 소켓 시스템 콜 시점, cgroup 부착

socket 훅은 앞의 두 훅과 본질적으로 다른 자리에 있습니다. 이 훅은 보통 cgroup v2 에 부착되며, 애플리케이션이 `connect()`, `sendmsg()`, `recvmsg()` 같은 소켓 시스템 콜을 호출할 때 트리거됩니다. 패킷이 만들어지기 전 단계, 즉 소켓이 "어디로 보낼지" 를 결정하는 시점에 끼어든다는 뜻입니다.

socket 훅의 컨텍스트는 `bpf_sock_addr` 같은 구조로, 소켓이 결정한 목적지 주소·포트 같은 메타데이터에 접근합니다. 패킷의 바이트가 아니라 "이 소켓이 향하는 목적지가 무엇인가" 를 본다는 점에서 앞의 두 훅과 시야가 다릅니다.

socket 훅의 처리 권한도 다른 훅과 다릅니다. 패킷 헤더를 가로채 재작성하는 NAT 가 아니라, 시스템 콜이 들고 있는 목적지 주소·포트 자체를 그 자리에서 수정하는 destination rewrite 모델입니다. 결과적으로 그 다음에 만들어지는 패킷은 처음부터 새 목적지를 향해 발사됩니다.

Cilium 은 이 자리를 socket-LB 에 활용합니다. Pod 가 `connect(ClusterIP)` 를 부르는 순간 백엔드 Pod IP 로 목적지가 바꿔치기되어, 이후 패킷 경로 어디에서도 별도의 NAT 변환이 필요하지 않게 됩니다. tc 나 XDP 에서 NAT 를 수행하면 그 연결에 속한 패킷마다 BPF 프로그램을 실행해 헤더를 재작성해야 하지만, socket 훅은 연결 수립 시점에 딱 한 번 목적지를 바꾸는 것으로 끝나므로 트래픽이 많아질수록 비용 차이가 커집니다. 코드 수준의 흐름은 이후 코드 분석 세션(`bpf/bpf_sock.c`)에서 다룹니다.

### "패킷 NAT" 가 아닌 destination rewrite

세 훅 중 socket 만은 "패킷을 가로채 헤더를 바꾼다" 는 일반적인 NAT 멘탈 모델로 생각하면 어긋납니다. XDP 와 tc 는 이미 만들어진 패킷의 흐름에 끼어들어 헤더를 검사·수정·전달하는 모델이지만, socket 훅은 패킷이 만들어지기 전에 소켓이 보고 있는 목적지 자체를 수정합니다. 따라서 socket-LB 가 적용된 흐름은 "VIP 패킷이 노드 어딘가에서 백엔드 IP 로 NAT 되었다" 가 아니라 "처음부터 백엔드 IP 로 향하는 패킷이 만들어졌다" 로 읽어야 합니다. 이 차이는 이후 코드 세션에서 BPF C 프로그램의 동작을 추적할 때 다시 나타납니다.

## 마무리

### 세 훅, 세 책임

세 훅의 부착 위치 차이는 Cilium 이 같은 패킷에 대해 서로 다른 일을 서로 다른 자리에 분배할 수 있게 만듭니다. 이 분배는 본 시리즈의 이후 세션 전반에서 반복적으로 나타납니다.

- XDP 자리에서는 패킷이 스택에 진입하기 전에 부하 분산 결정을 끝냅니다.
- tc 자리에서는 `sk_buff` 단위로 보안 정책을 집행합니다.
- socket 자리에서는 패킷을 만들기 전에 ClusterIP 를 백엔드 IP 로 바꿔치기합니다.

### 본 세션이 답하지 않은 것

이 문서는 세 훅의 부착 위치와 처리 권한 차이까지만 다뤘고, 다음 항목들은 의도적으로 미뤘습니다. kprobe, uprobe, tracepoint, cgroup 의 다른 변형, LSM 같은 다른 유형의 eBPF 훅은 0007 의 한 줄 언급에서 더 나아가지 않았습니다. tc 훅과 Envoy 를 연결하는 L7 리다이렉트, BPF map 의 실제 키 구조, 각 훅에 부착된 BPF C 프로그램의 코드 경로도 본 세션의 범위 밖입니다. Cilium 의 컴포넌트가 어떻게 이 훅들을 관리하는지는 다음 세션의 주제입니다.

### 다음 다리 — Cilium 아키텍처 개요

훅이 어디에 끼어드는지가 정해졌으니, 다음 질문은 "누가 그 훅에 BPF 프로그램을 부착하고 누가 BPF map 을 갱신하는가" 입니다. 다음 세션은 Cilium 의 컴포넌트 토폴로지(Agent, Operator, Hubble, CNI)를 클러스터 다이어그램 위에 배치하면서, 각 컴포넌트가 어느 자리의 어느 책임을 담당하는지를 정리합니다.
