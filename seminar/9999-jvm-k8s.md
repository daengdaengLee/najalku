# JVM 메모리는 왜 부하 후에도 줄지 않는가 — k8s 환경의 진단과 대응

## 문제

이 문서는 Kotlin과 Ktor로 작성된 서버 애플리케이션을 k8s에 배포해 부하 시험을 진행하던 중 관측된 메모리 거동에서 출발합니다. 정상 부하로 상정한 50 RPS에서는 CPU 사용률이 35% 내외, 메모리 사용률이 50% 내외를 유지하며 모든 요청을 정상 처리했습니다. 최대 부하로 상정한 100 RPS에서는 초반에 CPU가 솟구쳐 Pod가 scale out 되었다가, 이후 CPU가 45% 내외로 안정되면서 다시 최소 replica로 scale in 되었습니다.

문제는 메모리였습니다. 100 RPS 시험 동안 한 Pod의 메모리 사용률이 80%까지 올라간 뒤, 부하가 사라져도 내려오지 않았습니다. CPU는 부하를 따라 오르내리며 스케일이 정상 동작한 반면, 메모리만 높은 수준에 고정된 것입니다. 이 현상은 세 가지 질문을 낳습니다. 이것이 메모리 누수인지, 부하가 계속되면 OOMKill로 이어지는지, 그리고 CPU 사용률만으로 스케일을 제어해야 하는 환경에서 어떻게 대응해야 하는지입니다.

## 핵심 한 줄

메모리가 내려오지 않는 것은 누수가 아니라 JVM과 이 애플리케이션 스택의 구조적 거동입니다. 메모리는 순간순간 변하는 흐름이 아니라 한 번 확보하면 쌓여 남는 재고에 가깝기 때문입니다. 실제 위험은 상한이 없는 native 영역이 메모리 limit을 넘어 cgroup OOMKill로 이어지는 것이며, CPU 사용률만 보는 HPA는 이 위험을 방어하지 못합니다. 따라서 대응의 축은 스케일링 신호와 메모리 안전을 분리하는 것입니다.

## 배경 — 애플리케이션과 환경

이 문서가 다루는 대상은 Kotlin과 Ktor로 작성된 서버 애플리케이션이며, 외부 연동으로 다음을 사용합니다.

- DynamoDB
- Secrets Manager
- Parameter Store
- MSK

이 애플리케이션은 k8s에 다음과 같은 Pod 리소스로 배포되어 있습니다.

```yaml
resources:
  requests:
    cpu: 350m
    memory: 750Mi
  limits:
    cpu: 700m
    memory: 750Mi
```

메모리는 request와 limit을 같은 값(750Mi)으로 설정했습니다. 다만 CPU는 request 350m와 limit 700m으로 서로 다르므로, 이 Pod의 QoS 등급 자체는 Guaranteed가 아니라 Burstable입니다. QoS는 리소스별로 매겨지는 것이 아니라 Pod 단위로 정해지며, Guaranteed가 되려면 모든 컨테이너에서 CPU와 메모리가 모두 request와 limit이 일치해야 하기 때문입니다. 그럼에도 메모리만 두 값을 맞춘 데에는 이유가 있습니다. 메모리 사용량이 limit에 도달하면 컨테이너가 종료되는데, 이 Pod는 limit과 request가 같으므로 사용량이 request를 넘어선 상태로 존재할 수 없습니다. 그 결과 노드 메모리가 부족해 kubelet이 Pod를 축출할 때, 사용량이 request를 초과한 Pod가 먼저 축출되는 그 대상 그룹에 이 Pod는 들지 않습니다. 따라서 메모리 request와 limit을 일치시킨 것은 QoS 등급과 무관하게 노드 압박 상황에서의 축출 위험을 낮추는 효과가 있습니다.

오토스케일링은 CPU 사용률만으로 제어합니다.

```yaml
minReplicas: 3
maxReplicas: 10
metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
```

원래는 메모리 사용률도 HPA 조건에 두었으나 제거했습니다. 그 이유는 뒤에서 메커니즘과 함께 설명합니다. 한편 JVM은 힙 크기를 컨테이너 메모리에 비례해 잡도록 `-XX:MaxRAMPercentage=60`을 적용했습니다. 그 결과 힙 상한은 750Mi의 60%인 약 450Mi이며, 이 값이 이후 분석 전반의 기준선이 됩니다.

## 한 그림으로 보는 메모리 거동

부하 시험에서 관측된 CPU와 메모리의 거동을 시간 축 위에 함께 그리면 둘의 성격 차이가 한눈에 드러납니다.

```
부하   ▁▁▁▁████████████████▁▁▁▁▁▁▁▁▁▁▁▁   시간 →
           ↑               ↑
        부하 인가         부하 종료

 ├─ CPU     부하 구간에만 높음 → 종료와 함께 즉시 하강
 │            └ HPA: scale out 후 scale in 정상 동작
 │
 └─ 메모리   부하 구간에 상승 → 종료 후에도 그대로 유지
              └ 한 번 오른 값이 내려오지 않음 (흐름이 아닌 재고)
```

CPU는 순간순간의 작업량을 반영하므로 부하가 사라지면 즉시 하강하고, 그에 맞춰 HPA가 Pod 수를 늘렸다가 다시 줄입니다. 그러나 메모리는 한 번 확보하면 누적되어 남는 재고에 가깝습니다. 그 결과 부하가 빠진 뒤에도 메모리만 높은 수준에 머무릅니다. 이 비대칭이 이 문서가 풀어야 할 핵심이며, 왜 메모리가 내려오지 않는지, 그것이 위험한지, 그리고 CPU만으로 스케일하는 환경에서 어떻게 대응할지를 차례로 살펴봅니다.

## 메커니즘

앞 그림이 보여 준 메모리의 거동을 메커니즘 수준에서 분해합니다. 먼저 메모리가 내려오지 않는 이유와 그것이 OOMKill로 이어지는 경계를 살펴보고, 이어서 그 이해에서 따라 나오는 대응 — 스케일링 신호의 분리, CPU 요청량 산정, 가비지 컬렉터 선택 — 을 다룹니다.

### 메모리가 줄어들지 않는 이유

메모리가 내려오지 않는 첫 번째 원인은 JVM의 힙 관리 방식입니다. JVM은 부하가 몰리는 동안 객체 할당이 폭증하면 힙을 `MaxRAMPercentage`가 정한 상한까지 키워 OS로부터 메모리를 commit합니다. 이렇게 한 번 commit한 힙은 기본적으로 OS에 다시 반환되지 않습니다. 더 중요한 점은 힙 축소와 반환을 판단하는 계기가 가비지 컬렉션인데, 가비지 컬렉션은 객체 할당 압력에 의해 트리거된다는 사실입니다. 따라서 부하가 사라져 할당이 멈추면 가비지 컬렉션이 돌지 않고, 그 결과 힙에 빈 공간이 많으니 OS에 돌려주자는 판단 자체가 일어나지 않습니다. 즉 부하가 빠진 뒤에도 commit된 힙은 그대로 남습니다.

힙 바깥에서도 같은 일이 벌어집니다. 이 애플리케이션은 Ktor의 Netty 엔진과 AWS SDK의 비동기 클라이언트를 통해 다량의 direct buffer를 사용합니다. Netty는 이 buffer를 풀(pool) 형태로 관리하면서 부하에 맞춰 풀을 키운 뒤, 부하가 줄어도 풀을 적극적으로 반납하지 않습니다. 이 direct buffer는 힙 바깥의 native 메모리이므로 `MaxRAMPercentage`의 통제도 받지 않습니다. 그러므로 힙과 무관하게 줄어들지 않는 영역이 따로 존재합니다.

80%라는 수치를 분해하면 이 거동이 자연스럽다는 점이 드러납니다. 메모리 limit 750Mi의 80%는 600Mi이며, 이는 다음과 같이 구성됩니다.

```
750Mi (limit)
 600Mi ─┬─ 비힙(native) ≈ 150Mi
        │    메타스페이스 · 스레드 스택 · 코드 캐시 · GC 구조체 · direct buffer
        │
 450Mi ─┴─ 힙 commit ≈ 450Mi   (= 750Mi × MaxRAMPercentage 60%)
        │
   0 ───┘
```

따라서 600Mi는 힙이 상한까지 commit된 상태에 통상적인 native 사용량이 더해진 결과로 볼 수 있으며, 그 자체로 이상 징후라고 단정하기는 어렵습니다.

### OOMKill 의 경계

그렇다면 부하가 계속 몰릴 때 이 Pod가 OOMKill로 강제 종료될 수 있는지가 다음 질문입니다. 결론적으로 가능하지만, 그 경로를 정확히 이해하는 것이 중요합니다. JVM 힙과 컨테이너가 인식하는 메모리는 서로 다른 층위이기 때문입니다.

힙은 `MaxRAMPercentage` 60%에 의해 450Mi로 상한이 고정되어 있습니다. 따라서 살아있는 객체가 힙을 가득 채우면 JVM은 자바 수준의 `OutOfMemoryError`를 던지는데, 이는 cgroup이 프로세스를 죽이는 OOMKill과는 다른 사건입니다. 게다가 힙은 450Mi를 넘지 못하므로 힙만으로는 750Mi 한계에 도달하지 못합니다.

실제 위험은 상한이 없는 native 영역에 있습니다.

```
750Mi (limit) ─┬──────────────  넘으면 cgroup OOMKill (종료 코드 137)
               │  native 영역 (예산 ≈ 300Mi)   ← MaxRAMPercentage 통제 밖
               │   Netty direct buffer · 스레드 스택 · 메타스페이스 · 코드 캐시
 450Mi ────────┼──────────────  힙 상한 (750Mi × 60%)
               │  Java heap
               │   가득 차면 OutOfMemoryError (cgroup OOMKill 아님)
   0 ──────────┴──────────────
```

부하가 지속되어 동시성이 올라가면 Netty direct buffer 풀과 스레드 수가 늘어나고, 힙 commit 450Mi에 이 native 사용량이 더해져 750Mi를 넘는 순간 커널이 프로세스를 종료합니다. 그러므로 이 구성에서 OOMKill이 발생한다면 그 원인은 거의 힙이 아니라 native 영역의 증가입니다.

이 지점에서 한 가지 사각지대가 드러납니다. HPA는 CPU 사용률만으로 스케일을 결정하므로, CPU가 목표치 아래로 안정된 상태에서 메모리만 차오르면 scale out이 트리거되지 않습니다. 즉 스케일을 결정하는 신호인 CPU와 실패를 일으키는 요인인 메모리가 분리되어 있어, 메모리 증가를 HPA가 방어해 주지 못합니다.

### 스케일링 신호와 메모리 안전의 분리

앞서 메모리를 HPA 조건에서 제거했다고 밝혔는데, 그 이유가 여기에 있습니다. 메모리를 HPA에 넣으면 scale in이 영영 일어나지 않는 문제가 생깁니다. HPA는 여러 메트릭이 있을 때 각 메트릭이 요구하는 replica 수를 따로 계산한 뒤 그중 최댓값을 채택합니다. JVM 메모리는 한 번 오르면 내려오지 않으므로 메모리 메트릭이 요구하는 replica 수는 계속 높게 유지되고, CPU가 줄이라고 계산해도 메모리가 유지하라고 계산하는 쪽이 채택되어 Pod 수가 높은 상태에 고정됩니다. 부하가 사라져도 최대치까지 늘어난 Pod가 줄지 않는 것입니다. 따라서 메모리는 스케일링 신호로 적합하지 않습니다.

여기서 얻는 원칙은 스케일링 신호와 메모리 안전을 별개의 문제로 다루어야 한다는 것입니다. 스케일링은 부하를 따라 오르내리는 신호로 제어하고, 메모리 안전은 사이징과 native 영역 제어로 따로 보장합니다. 운영 제약상 CPU 사용률만으로 스케일을 제어해야 한다면, CPU를 부하의 대리 지표로 삼아 Pod 수를 적절히 늘림으로써 간접적으로 메모리를 분산할 수 있습니다. Pod가 많아지면 Pod당 처리량이 줄고, 그 결과 Pod당 native 메모리 누적도 줄기 때문입니다.

### CPU 요청량 산정

실측값을 환산하면 요청량(request) 설정에서 개선의 여지가 드러납니다. HPA의 사용률은 request 대비로 계산되므로, 실제 코어 사용량은 사용률에 request를 곱한 값입니다. 정상 부하 50 RPS에서 Pod당 약 122m, 최대 부하 100 RPS에서 약 157m를 사용했는데, 이는 설정된 request 350m의 절반에도 못 미칩니다. 즉 request가 실제 사용량의 약 두 배로 부풀려져 있었고, 그 결과 100 RPS에서도 사용률이 45%로만 읽혀 부당하게 scale in이 일어났습니다.

request를 실측에 맞춰 약 200m로 낮추면 같은 부하가 더 높은 사용률로 읽혀 HPA가 Pod 수를 적절히 유지합니다. 이렇게 하면 100 RPS는 약 6~7개의 Pod로 수렴하며, 이는 정상으로 확인된 50 RPS 시점의 Pod당 처리량과 같은 수준이어서 Pod당 메모리도 안전권에 머무릅니다. 구체적인 모델과 평형 Pod 수는 부록에 정리합니다.

다만 CPU는 메모리의 완전한 대리 지표가 아닙니다. CPU 기반 스케일링은 메모리가 부하에 비례할 때만 유효하며, 부하와 무관하게 시간에 따라 자라는 native 누수나 짧은 스파이크가 남긴 메모리에는 대응하지 못합니다. 따라서 CPU 조정은 보조 수단이고, 메모리 안전의 본질은 이어서 설명할 가비지 컬렉터 선택과 사이징에 있습니다.

### GC 와 CPU limit 의 상호작용

메모리를 실제로 회수하게 만들려면 가비지 컬렉터의 선택과 거동을 이해해야 합니다. 컨테이너 인식 JVM은 CPU limit으로부터 가용 프로세서 수를 계산하는데, 그 값은 올림 연산을 따릅니다. CPU limit 700m은 올림하여 1로, 1000m 역시 올림하여 1로 인식됩니다. 따라서 limit을 700m에서 1000m으로 올려도 JVM이 보는 코어 수는 1로 동일하며, 가비지 컬렉터 종류나 스레드 풀 크기는 바뀌지 않고 다만 throttle 여유만 늘어납니다.

CPU limit을 2000m으로 올리면 가용 프로세서가 2로 인식됩니다. 그러나 이것만으로 G1 가비지 컬렉터가 자동 선택되지는 않습니다. JVM이 G1을 기본값으로 고르는 server-class 판정은 프로세서 2개 이상과 메모리 1792MB 이상을 동시에 요구하는데, 이 Pod의 메모리 limit 750Mi는 그 문턱에 한참 못 미치기 때문입니다. 그러므로 이 구성에서 G1을 사용하려면 limit과 무관하게 `-XX:+UseG1GC`를 명시해야 합니다. 또한 프로세서를 2로 인식시키면 코루틴과 Netty, AWS SDK의 스레드 풀이 대략 두 배로 늘어 native 메모리가 오히려 증가하므로, 메모리 안전 관점에서는 역효과가 날 수 있습니다.

G1을 명시적으로 활성화하면 유휴 구간에 주기적으로 가비지 컬렉션을 돌려 사용하지 않는 힙을 OS에 반환하게 만들 수 있습니다. `-XX:G1PeriodicGCInterval`이 그 간격을 정합니다. 이를 통해 부하 스파이크 사이에 메모리를 회수해 두면, 이미 많은 메모리를 쥔 Pod가 다음 스파이크에서 먼저 한계에 부딪히는 상황을 완화할 수 있습니다.

## 정상 거동과 누수의 구분

지금까지의 설명은 이 현상이 정상 거동이라는 전제에 기대고 있으므로, 마지막으로 그것이 정상 거동인지 실제 누수인지 가르는 방법이 필요합니다. 핵심은 부하를 장시간 일정하게 유지하면서 메모리 추이를 관찰하는 것입니다. 워밍업 이후 메모리가 일정 수준에서 평평하게 유지되면 commit된 메모리를 쥐고 있을 뿐인 정상 거동이고, 여러 시간에 걸쳐 꾸준히 우상향하면 누수를 의심해야 합니다.

누수가 의심되면 native memory tracking으로 어느 영역이 자라는지 좁혀 들어갑니다. 힙 commit이 계속 늘면 힙 누수, direct buffer가 늘면 Netty 버퍼 누수나 풀 비대, 스레드 수가 늘면 무제한으로 생성되는 스레드 풀, 메타스페이스가 늘면 클래스로더 누수가 원인입니다. 또한 Pod가 실제로 종료되었다면 종료 코드가 단서가 됩니다. 종료 코드 137에 자바 예외 로그가 없으면 native 초과로 인한 cgroup OOMKill이고, 로그에 `OutOfMemoryError`가 남아 있으면 힙이나 특정 native 상한에 도달한 경우입니다. 영역별 진단 기준은 부록에 정리합니다.

## 마무리

정리하면, 부하 후 메모리가 내려오지 않는 현상은 JVM이 commit한 힙과 Netty가 확보한 direct buffer를 반환하지 않는 구조적 거동이며, 그 자체로는 누수가 아닐 가능성이 높습니다. 그러나 상한이 없는 native 영역이 메모리 limit을 넘으면 cgroup OOMKill로 이어질 수 있고, CPU만 보는 HPA는 이를 방어하지 못합니다. 따라서 대응의 핵심은 스케일링 신호와 메모리 안전을 분리하여, 스케일링은 부하를 따르는 CPU 신호로 제어하고 메모리 안전은 native 영역 제어와 실측 기반 사이징으로 보장하는 것입니다. CPU 요청량과 limit 조정은 그 위에 얹는 보조 수단이며, 어떤 조치든 적용 전후로 장시간 부하 시험과 native memory tracking을 통해 그 효과를 검증해야 합니다.

## 부록

### CPU 사용량 모델과 평형 Pod 수

실측 두 점에서 Pod당 CPU 사용량을 부하에 대한 선형 모델로 근사하면 다음과 같습니다. baseline은 부하와 무관한 가비지 컬렉션과 백그라운드 스레드, JIT 컴파일의 몫입니다.

```
Pod당 CPU ≈ 87.5m + 2.1m × (Pod당 RPS)
```

request 200m, target 60%를 적용했을 때의 평형 Pod 수 추정은 다음과 같습니다.

| 부하 | Pod 수 | Pod당 RPS | Pod당 CPU | 사용률 |
| --- | --- | --- | --- | --- |
| 50 RPS | ~4 | ~12 | ~114m | ~57% |
| 100 RPS | ~6–7 | ~15 | ~120m | ~60% |
| 유휴 | 3 (min) | 0 | ~87.5m | ~44% |

baseline 87.5m은 request 200m의 약 44%를 차지합니다. HPA는 사용률과 target의 비가 기본 허용 오차(tolerance) 10% 밴드를 벗어나야 replica 수를 조정하므로, 유휴 상태에서 최소 replica까지 scale in이 일어나려면 사용률을 target으로 나눈 값이 0.9보다 작아야 합니다. 이 조건을 유휴 사용률 44%에 대입하면 target의 하한은 약 49%이고, target을 60%로 유지한다면 request의 하한은 약 160m입니다. 이보다 target이나 request를 더 낮추면 유휴 Pod의 사용률이 허용 오차 밴드 안으로 들어와 scale in이 멈추고, 한 번 늘어난 Pod 수가 줄지 않아 maxReplicas 근처에 고정됩니다. 이것이 사실상의 하한선입니다.

### native 증가 위치별 진단

native memory tracking에서 어느 영역이 자라는지에 따라 원인과 대응이 갈립니다.

| 증가 영역 | 의심 원인 | 대응 방향 |
| --- | --- | --- |
| heap committed | 힙 누수 | 힙 덤프로 retained 객체 추적 |
| Direct / Netty | 버퍼 누수 · 풀 비대 | `MaxDirectMemorySize` 상한, allocator 점검 |
| Thread | 무제한 스레드 풀 | 해당 풀에 상한 설정 |
| Metaspace | 클래스로더 누수 | 동적 클래스 생성 점검 |
