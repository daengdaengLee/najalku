# eBPF — 커널 내부에서 안전하게 실행되는 프로그램과 공유 map

이 문서는 Cilium 시리즈의 첫 번째 자료입니다. Cilium 이 어떤 컴포넌트로 구성되고 어떻게 동작하는지는 다음 세션부터 본격적으로 살펴보고, 여기서는 Cilium 이 기반으로 삼는 기술인 eBPF 그 자체를 다룹니다. 이전 세미나(0006 kube-proxy iptables) §5 는 KUBE-* 체인의 구조적 한계를 나열하고 eBPF 기반 데이터플레인을 그 다음 방향으로 지목했습니다. 이 문서는 그 물음에 대한 답입니다.

## 문제 — iptables 룰이 부딪힌 한계

kube-proxy iptables 모드는 Service 하나당 KUBE-SVC-* 체인 하나와 엔드포인트 수만큼의 KUBE-SEP-* 체인을 netfilter 규칙으로 표현합니다. 클러스터 규모가 커질수록 이 구조는 세 가지 방향에서 벽에 부딪힙니다.

- 패킷당 매칭 비용 O(n): Service 와 엔드포인트가 늘어날수록 노드당 iptables 룰 수가 선형으로 증가하고, netfilter 는 체인 안의 룰을 위에서 아래로 순회하므로, 패킷 한 개를 처리하는 비용도 룰 수에 비례해 늘어납니다.
- 변경 시 풀 재기록 비용: 부하 분산을 `-m statistic --probability` 의 역순 동전 던지기로 구현하기 때문에, 엔드포인트가 바뀔 때마다 균등을 유지하려면 체인 전체를 다시 써야 합니다. partial sync 가 있어도 `FullSyncPeriod` 마다 또는 큰 변경 시점에는 결국 노드 전체 룰 셋을 재생성·재적용합니다.
- 디버깅 난이도: `iptables-save -t nat | grep KUBE-` 출력에서 KUBE-SERVICES → KUBE-SVC-* → KUBE-SEP-* 점프 경로를 사람이 손으로 따라가야 합니다.

```
패킷 ──▶ KUBE-SERVICES
              │
              ├──▶ KUBE-SVC-AAAA ──▶ KUBE-SEP-001 (Pod A1)
              │                  └──▶ KUBE-SEP-002 (Pod A2)
              │
              ├──▶ KUBE-SVC-BBBB ──▶ KUBE-SEP-003 (Pod B1)
              │                  └──▶ KUBE-SEP-004 (Pod B2)
              │
              └──▶ ...   (Service × 엔드포인트 수만큼 선형 누적)
```

문제의 뿌리는 구조 자체에 있습니다. KUBE-* 체인은 netfilter 라는 고정된 커널 경로 위에 덧씌운 규칙 집합이고, 규칙을 수정하는 것 외에 그 경로를 바꿀 방법이 없습니다. 한계를 피하려면 체인을 다듬는 것이 아니라 데이터플레인 자체를 갈아끼워야 합니다.

## 핵심 한 줄

eBPF(extended Berkeley Packet Filter)는 커널 내부에서 안전하게 실행되는 사용자 정의 프로그램과, 그 프로그램이 사용자공간과 데이터를 공유하는 자료구조(BPF map)의 조합입니다. 커널 소스를 수정하거나 커널 모듈을 별도로 컴파일하지 않고도 커널 경로 중간에 프로그램을 끼워 넣을 수 있다는 것이 핵심입니다.

## 메커니즘 — 무엇이 안전을 보장하고 무엇이 성능을 만드는가

### hook point — 어디에 끼어드는가

eBPF 프로그램은 커널 안의 특정 지점(hook point)에 부착되어 그 지점을 지나는 이벤트가 발생할 때마다 호출됩니다. Cilium 이 사용하는 지점은 크게 세 곳입니다.

- XDP(eXpress Data Path): NIC 드라이버가 패킷을 받아들이는 가장 이른 지점입니다. 패킷이 아직 `sk_buff`(소켓 버퍼) 구조체가 되기 전에 실행되므로 처리 비용이 가장 낮습니다.
- tc ingress/egress: 트래픽 제어(traffic control) 계층에서 동작합니다. `sk_buff` 단위로 처리하며, XDP 보다는 늦지만 네트워크 스택 대부분이 실행되기 전입니다.
- socket: 소켓 시스템 콜(`connect()`, `sendmsg()` 등) 시점에 끼어듭니다. 패킷이 만들어지기 전, 소켓이 목적지를 결정하는 단계에서 개입합니다.

```
   ────── 수신 경로 ──────          ────── 송신 경로 ──────

        NIC                              애플리케이션
         │                                   │
         ▼                                   ▼
    [XDP hook]                          [socket hook]
         │                                   │
         ▼                                   ▼
    [tc ingress]                          소켓 계층
         │                                   │
         ▼                                   ▼
      네트워크 스택                       네트워크 스택
         │                                   │
         ▼                                   ▼
       소켓 계층                          [tc egress]
         │                                   │
         ▼                                   ▼
      애플리케이션                            NIC

    [XDP hook]    NIC 드라이버 직후, sk_buff 생성 전
    [tc ingress]  수신 측 sk_buff 단위 처리
    [tc egress]   송신 측 sk_buff 단위 처리
    [socket hook] connect() / sendmsg() 등 시스템 콜 시점
```

kprobe, uprobe, tracepoint, cgroup, LSM 등 다른 유형의 hook 도 존재하지만, 이 시리즈에서는 위 세 종류만 다룹니다.

### verifier — 안전성 정적 검증

임의의 코드를 커널 안에서 실행한다는 것은 곧 커널 전체를 위험에 노출시킬 수 있다는 뜻입니다. eBPF 는 이 문제를 verifier 로 해결합니다. BPF 프로그램을 커널에 로드하는 순간 verifier 가 바이트코드를 정적으로 분석해 두 가지를 보장합니다. 첫째, 프로그램이 반드시 종료합니다 — 무한루프가 없습니다. 둘째, 허용된 메모리 범위 밖에 접근하지 않습니다. 이 검증을 통과하지 못한 프로그램은 아예 로드가 거부됩니다. 따라서 커널 모듈과 달리 검증되지 않은 BPF 프로그램이 커널을 손상시키는 일이 구조적으로 차단됩니다.

### JIT — 인터프리터에서 네이티브 명령어로

verifier 를 통과한 BPF 바이트코드는 JIT 컴파일러를 거쳐 해당 아키텍처(x86_64, ARM64 등)의 네이티브 명령어로 변환됩니다. 그 결과 BPF 프로그램은 커널 함수와 동일한 속도로 실행됩니다. netfilter 가 KUBE-* 체인의 룰을 위에서 아래로 순회하는 방식과 달리, JIT 컴파일된 BPF 프로그램은 직접 실행 코드이기 때문에 패킷당 처리 비용이 룰 수와 무관하게 고정됩니다.

### BPF map — 커널과 사용자공간이 공유하는 자료구조

BPF 프로그램 자체는 매 호출이 끝나면 종료하므로 호출 간에 상태를 보유하지 못합니다. 그래서 BPF 프로그램은 여러 호출에 걸쳐 상태를 유지하고 사용자공간과 데이터를 주고받기 위해 BPF map 이라는 커널-사용자공간 공유 자료구조를 사용합니다. BPF map 은 키-값 저장소로, 사용자공간의 제어 프로그램이 항목을 삽입·갱신하면 커널에서 실행 중인 BPF 프로그램이 이를 조회할 수 있습니다. 반대 방향도 동일합니다. 이 양방향 공유 덕분에 사용자공간이 커널 코드의 동작을 동적으로 제어할 수 있습니다.

```
사용자공간                                    커널공간
──────────────────────                        ──────────────────────

  제어 프로그램                                  BPF 프로그램
  (Cilium Agent 등)                              (hook 에서 실행 중)
         │                                              │
         │ bpf_map_update_elem()                        │
         └──────────────────────▶ BPF map ◀─────────────┘
                                 (키-값 저장소)  bpf_map_lookup_elem()
                                                  (커널 코드 내부에서 조회)
```

map 의 구체적인 키 구조, Cilium 에서 어떤 유형의 map 이 어떻게 쓰이는지는 다음 세션들에서 Service LB 와 NetworkPolicy 를 다루며 하나씩 살펴봅니다.

## 한 그림으로 보는 eBPF

hook point, verifier, JIT, BPF map 네 가지를 한 장의 그림으로 묶으면 eBPF 의 전체 구조가 드러납니다.

```
사용자공간                                 커널공간
──────────────────────────                 ──────────────────────────────────────

  BPF 소스코드 (.c)
        │
   clang 컴파일
        │
  바이트코드                                 ┌─── verifier ───────────────────────┐
        │                                 │  무한루프 · 잘못된 접근 차단             │
        │ bpf() 로드 ─────────────────────▶│                                    │
        │                                 │            검증 통과                 │
        │                                 │                │                   │
        │                                 │                ▼                   │
        │                                 │          JIT 컴파일                  │
        │                                 │      바이트코드 → 네이티브 명령어         │
        │                                 │                │                   │
        │                                 │                ▼                   │
        │                                 │         hook 지점 설치               │
        │                                 │         XDP │ tc │ socket          │
        │                                 └────────────────┼───────────────────┘
        │                                                  │
        │                    네트워크 패킷 ─────────────────▶ │
        │                    소켓 시스템 콜 ────────────────▶ │
        │                                                  ▼
        │                                          BPF 프로그램 실행
        │                                                  │
        │                                           BPF map 조회
        │                                                  │
  제어 프로그램                                               ▼
        │                                  ┌─────────────────────────────┐
        │ bpf_map_update_elem() ──────────▶│         BPF map             │
        │                                  │        (키-값 저장소)          │
        └◀─ bpf_map_lookup_elem() ─────────│                             │
                                           └─────────────────────────────┘
                                                           │
                                         패킷 허용 · 드롭 · 주소 변환 · 전달
```

사용자공간에서 로드한 BPF 프로그램은 verifier 를 통해 안전성을 보장받고, JIT 를 거쳐 네이티브 속도로 hook 지점에 설치됩니다. 그 뒤로는 패킷이나 소켓 시스템 콜이 hook 지점을 지날 때마다 BPF 프로그램이 실행되고, BPF map 을 통해 사용자공간의 제어 프로그램과 데이터를 주고받습니다. KUBE-* 체인처럼 고정된 커널 경로에 선형으로 규칙을 쌓는 것이 아니라, 커널 경로 중간에 프로그램 자체를 끼워 넣는 구조입니다.

## 마무리

### iptables 한계에 대한 응답

앞서 정리한 세 가지 한계 각각에 eBPF 는 다음과 같이 응답합니다.

- 패킷당 매칭 비용 O(n): 룰을 순회하는 대신 BPF map 을 조회하고, JIT 컴파일된 BPF 프로그램이 직접 실행 코드로 동작합니다. map 조회는 규칙 수와 무관하게 O(1) 이고 패킷당 처리 비용도 고정됩니다.
- 변경 시 풀 재기록 비용: 엔드포인트 목록을 BPF map 에 저장하고 BPF 프로그램이 이를 조회해 선택합니다. 엔드포인트나 Service 가 바뀌면 map 항목만 삽입·삭제·갱신하면 되므로, 전체 룰 셋을 재생성·재적용하는 풀 동기화 자체가 필요하지 않습니다.
- 디버깅 난이도: eBPF 는 커널 이벤트를 사용자공간으로 내보내는 관찰성 hook 을 제공합니다. 구체적인 메커니즘은 Hubble 을 다루는 이후 세션에서 살펴봅니다.

### 본 세션이 답하지 않은 것

이 문서는 eBPF 의 구성 요소를 개별적으로 살펴봤을 뿐, Cilium 이 이 요소들을 어떻게 조립하는지는 다루지 않았습니다. BPF map 의 실제 키 구조, Cilium 이 사용하는 BPF C 프로그램의 작성 방식, Cilium Agent 가 k8s 이벤트를 BPF map 에 어떻게 반영하는지는 다음 세션들에서 하나씩 다룹니다.
